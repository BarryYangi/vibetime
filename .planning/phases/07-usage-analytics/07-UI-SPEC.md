## Usage Page Contract

### Route And Navigation

- Add `/usage` as a dedicated route.
- Place Usage immediately after History and before Settings in `Sidebar`.
- Include `/usage` in last-view persistence.
- Do not nest Usage inside History and do not add a History tab for Usage.
- Use the same `PageShell fluid` dashboard shape as History: `flex flex-col gap-5 py-7 sm:px-7 sm:py-8`.

### Header

The header uses the History pattern:

- Eyebrow: `Usage analytics` / `用量分析`
- H1: `Usage` / `用量`
- Right controls in this order: loading spinner, period tabs, optional filters, `Refresh Usage` button.
- Period tabs are exactly `7d`, `30d`, `90d`, `365d`, backed by the existing `HISTORY_PERIODS` values and localized with the same formatter as History.

### Controls

- Period selector: coss `Tabs`, same dimensions as History (`h-6`, 11.5px labels, tabular numerals).
- Agent filter: coss `Select`, options limited to `All`, `Claude Code`, `Codex`.
- Project filter: coss `Select`, populated only from usage rows linked to projects; no placeholder rows for Cursor/Gemini.
- Model filter: coss `Select`, populated from scanned usage models.
- Sidechain/subagent inclusion: coss `Switch` only if scanner data exposes `isSidechain` or equivalent. Default: included in project totals, filterable.
- Pricing refresh: the page attempts refresh on open, renders cached values first, then updates state after success/failure. The CTA manually triggers both usage scan and pricing refresh.

### Settings Control

Add a Settings row under App Startup or a new Usage section:

- Label: `Usage refresh frequency` / `用量刷新频率`.
- Control: coss `Select`.
- Options: `Every 15 minutes`, `Every 30 minutes`, `Every hour`, `Every 4 hours`.
- Default: `Every 30 minutes`.
- Behavior: a background scan starts after app launch; this setting controls subsequent scan cadence. Hook invocations never scan transcripts, fetch prices, or compute aggregates.

### Data States

- Loading: blank background shell for first route suspense; within loaded Usage, show a 14px muted line plus spinner without shifting dashboard layout.
- Empty: show the empty state copy above, no charts, and no Cursor/Gemini explanation beyond the Claude/Codex-only note.
- Partial pricing failure with cache: show metric values from cache, a muted `Using cached pricing.` status, and keep `Refresh Usage` enabled.
- Pricing failure with no usable cache/model price: show token totals, set cost fields to `Unknown`, and show `Pricing unavailable. Token totals are still shown.`
- Unassigned usage: never hide it. Show an audit row/card labeled `Unassigned usage` with count, tokens, cost if known, and attribution method summary.
- Error: show the error state copy in a `PageShell fluid` layout; do not open modal dialogs for passive load failures.

## Approved Visualization Inventory

Final charts must be selected from this inventory only after the required `data capability -> hook linkage -> user value -> visualization` mapping proves the needed fields. Unsupported charts render an unavailable panel with a concise reason; they must not show synthetic data.

| Visualization | Required Data | Contract |
|---------------|---------------|----------|
| Metric tiles | usage tokens, pricing status, period range | Four tiles: estimated cost, total tokens, cache hit rate, top model or agent/model mix |
| Daily usage trend | usage timestamp, token totals, estimated cost nullable, agent/model | ECharts stacked bar or area by day; switch metric between cost and tokens if both are available |
| Token breakdown | input, cached input, cache creation, output, reasoning | ECharts stacked horizontal bar or compact segmented bars; reasoning is displayed but not double-counted in cost |
| Project spend ranking | hook-linked project plus usage cost/tokens | Table or bar list by project; unavailable until project attribution exists |
| Model efficiency | model, output tokens, duration or turn count | Table first; chart only if duration linkage exists |
| Cost/time relationship | estimated cost and hook duration on the same turn/window | Scatter plot; unavailable without hook linkage |
| Trends by day/week/month | daily aggregates plus optional weekly/monthly rollup | Daily view is required; weekly/monthly summaries may be compact stat rows in MVP |
| Audit table | unknown model, unknown price, unassigned rows, attribution method | Dense coss `Table`; sortable by tokens, cost, model, project, confidence |

Minimum first usable dashboard after mapping:

1. Four metric tiles.
2. Daily usage trend.
3. Token breakdown.
4. Project/model/agent table.
5. Pricing/status/audit panel.

Cursor and Gemini are excluded from every Usage aggregate and legend in Phase 07.

## Interaction Contracts

- Renderer reads Usage data only through typed IPC; no renderer SQLite or filesystem access.
- Page open sequence: render cached DB/pricing state -> request pricing refresh -> request/observe background scan state -> recompute and refresh visible aggregates.
- Refresh CTA shows loading on the button, disables duplicate clicks, and preserves stale visible data until fresh data arrives or a failure state is known.
- Background scanner push events should update Usage only after the Usage view has loaded once, mirroring History's best-effort refresh pattern.
- Sorting in tables follows History: clickable headers with lucide up/down sort icons, 11px table headers, 13px body cells.
- Charts use existing ECharts init/dispose/resize hook pattern from `History.tsx`; no custom canvas/SVG chart engine.
- Tooltips are confined, use chart theme tokens, and must include whether cost is estimated or unknown when hovering cost values.
- Hover/focus states use existing coss classes and default macOS arrow cursor, matching the project decision in `index.css`.
- Cost values format as USD with two decimals for totals under $1,000; token values use compact localized number formatting; unknown cost displays text, not `$0.00`.

---

## Layout Contract

- Use no nested UI cards. Page sections are unframed layout bands; repeated panels are the only card-like containers.
- Dashboard panels use the existing History `DashboardPanel` visual contract: rounded-xl, `border-border/60`, `bg-card/40`, `shadow-sm`, internal `px-5 pt-4 pb-5`.
- Stat tiles match History density: 4-column grid at `md`, gap 8px, stable min-height, tabular numerics.
- Main two-column dashboard groups may use `xl:grid-cols-2`; dense tables span full width.
- Minimum app viewport remains `960px x 640px`; no mobile-specific contract is required for this Electron phase.
- Long project/model names truncate in table/list rows and expose full value in accessible title text or tooltip.
